# dev-skill · 完整提问模板（≥10 问）

启动新任务时用 `AskUserQuestion` 一次性问。**跟进 / 「直接改」 / < 20 行小修跳过**。

**规则**：
- 所有问题用 AskUserQuestion 工具（CLAUDE.md 铁律）
- 一次性问完，不要分多轮问（Claude Design 原文：「一轮集中提问通常就够」）
- 每组最多 4 个选项
- 第一选项打 `(Recommended)` 如果有明显最优

---

## 必问 4 条（任务核心）

### Q1 · 不变量
```
问题：这个改动的不变量是什么？哪些行为绝不能退化？
选项：
- 列在 PRD 里了（指路）
- 我来列一下：<具体行为>
- 没有强不变量（纯新功能）
- 先写再说 — 出问题再讨论（不推荐但可选）
```

### Q2 · 现有 pattern
```
问题：代码库里有相似功能的已实现吗？
选项：
- 有：<路径>，按它的 pattern 做
- 没有，你探索一下决定
- 以前有过但风格要换
- 故意要新风格，不参考现有
```

### Q3 · 验证标准
```
问题：哪些场景必须测，哪些可以跳过？
选项：
- 全路径测试 (happy + edge + error)
- 只测核心 happy path
- 只测 regression （已有功能不能坏）
- 手动测即可，暂不加自动化
```

### Q4 · 部署约束
```
问题：部署约束 / 上线节奏？
选项：
- 可独立上线
- 要和其他改动打包
- 需要 feature flag 灰度
- 只改本地，不部署
```

## 问题特化 4 条（根据任务补充）

### Q5 · 性能 vs 简洁
```
问题：性能关键还是简洁优先？
选项：
- 简洁优先（除非 profiling 证明热点）
- 性能关键（明确场景：<QPS/延迟>）
- 两个平衡
- 暂不考虑，后续再优化
```

### Q6 · 错误处理范围
```
问题：错误处理严格程度？
选项：
- 只在系统边界验证 (Recommended，全局指令默认)
- 每层都 defensive
- 用户输入层 + 外部 API 层
- 暂无需强处理，失败抛异常即可
```

### Q7 · 测试介质
```
问题：测试层次？
选项：
- 单元测试为主
- 集成测试为主（需要真实依赖）
- E2E 为主（用户视角）
- 三层都要
```

### Q8 · 文档更新
```
问题：需要更新哪些文档？
选项：
- CLAUDE.md（新增约定）
- README（用户指南）
- inline docstring（API）
- 都不用（代码自解释）
```

## 补充问项（共 10 个凑齐，以下按任务选）

### Q9 · 依赖新增
```
问题：这个任务会需要新依赖吗？
选项：
- 明确需要：<包名 + 理由>
- 探索后决定
- 用已有依赖实现
- 禁止新增依赖
```

### Q10 · 回滚策略
```
问题：如果上线后发现问题怎么回滚？
选项：
- 单 PR revert 即可
- 数据库 migration 需配套 down script
- Feature flag 关闭
- 灾难级：需要预先演练回滚流程
```

## Claude Design 原文对照

Claude Design 对设计任务的提问指南：
- 「总是确认起点和产品上下文 —— UI kit、设计系统、代码库等」
- 「启动新项目问至少 10 个问题」
- 「提问工具调用后结束回合 —— 不要 monologue」

我们映射到开发任务：
- 起点 = 现有代码 + 技术栈 + PRD
- ≥10 问 = 必问 4 + 特化 4 + 补充 2
- 提问后结束 = AskUserQuestion 调用后等待，不继续推理
