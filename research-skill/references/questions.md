# research-skill · 完整提问模板（≥10 问）

---

## 必问 4 条

### Q1 · 决策范围
```
问题：这次调研的终点是什么？
选项：
- 只需技术选型（给推荐即可）
- 选型 + 落地计划（要 PoC 代码 / 迁移步骤）
- 只需"了解一下这个领域"（无需决策）
- 验证一个假设（有预设答案，要我找证据）
```

### Q2 · 基线约束
```
问题：有哪些硬约束？
选项：
- 现有技术栈指定（必须兼容 <栈>）
- 时间窗口（<N> 周内要决策）
- 成本上限（<金额>）
- 团队熟悉度（只考虑团队会的技术）
```

### Q3 · 保真度
```
问题：要多详细的调研？
选项：
- 一页结论纸（5 分钟能读完）
- 标准报告（≥3 候选 + 决策矩阵 + 风险）
- 深度对比（包含 PoC snippet + 性能 benchmark）
- 学术级（包含论文 / 社区讨论 / 历史演进）
```

### Q4 · 变体数量
```
问题：要对比几个候选？哪些维度关键？
选项：
- 3 个标准候选（默认）
- 2 个对冲（A vs B 深度对比）
- 5+ 广度扫描（然后收敛）
- 你决定，列清单让我点掉
```

## 特化 4 条

### Q5 · 权威性要求
```
问题：引用来源的权威性要求？
选项：
- 官方文档优先（Recommended）
- 官方 + 社区实战（Reddit / HN / 博客）
- 同行公司的生产实践（Netflix / Airbnb tech blog）
- 学术论文（需严谨，花时间多）
```

### Q6 · 时效性
```
问题：多久的资料还算新鲜？
选项：
- 1 年内（快速演进领域，如 LLM / JS 生态）
- 3 年内（大多数领域）
- 5 年内（稳定领域，如数据库）
- 不限（历史性话题）
```

### Q7 · 风险偏好
```
问题：推荐倾向保守还是激进？
选项：
- 保守：验证过、成熟、大厂在用
- 平衡：最优工具，即使偏新（Recommended）
- 激进：前沿、bleeding edge、愿意试错
- 让我自己选，你列完利弊
```

### Q8 · 现有代码权重
```
问题：现有代码库对决策的权重？
选项：
- 必须兼容现有栈（最高权重）
- 优先但不绝对
- 参考即可，新栈也行
- 不相关（绿地项目）
```

## 补充 2 条凑 10

### Q9 · 输出形态
```
问题：除了主报告，还要什么？
选项：
- 只要 .md 报告
- + 决策矩阵（HTML 表格 / Excalidraw）
- + PoC snippet（每个候选的最小 demo）
- + Obsidian 长期笔记（结论沉淀）
```

### Q10 · 下游动作
```
问题：调研结束后下一步？
选项：
- 直接开工（切 dev-skill）
- 先做 PoC 验证再开工
- 需要再一次用户讨论会决策
- 暂时不动（只做信息收集）
```

## Claude Design 原文启示

> "总是确认起点和产品上下文——UI kit、设计系统、代码库等。没有就让用户附上。没有上下文开始设计一定做出烂设计，要避免。"

调研映射：
- 起点 = 现有技术栈 + 约束 + 团队背景
- 产品上下文 = 用户解决的业务问题（而不是"选库"本身）
- 没上下文就乱给推荐 = 调研失败
